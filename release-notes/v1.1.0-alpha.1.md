# pdfnative v1.1.0-alpha.1 — Release Notes

**Release date:** 2026-04-29
**Status:** Alpha pre-release on the path to v1.1.0 stable.
**Highlights:** Watermark auto-fit, Unicode ellipsis, column min/max
constraints, four new PDF decode filters in the parser, live version
widget for the docs site.
**Breaking:** Two intentional, opt-out-able changes — see the dedicated
section below.

---

## Why an alpha?

Two of the v1.1.0 milestone items — issue
[#28](https://github.com/Nizoka/pdfnative/issues/28) (PDF/A Latin font
embedding) and issue [#25](https://github.com/Nizoka/pdfnative/issues/25)
(full UAX #9 BiDi + multi-pass GSUB) — require atomic multi-file
object-graph rewrites and font-data rebuilds whose risk profile is
incompatible with shipping in the same iteration as smaller, fully-tested
changes. Rather than block the rest of the v1.1.0 surface behind those
epics, this alpha lands every Medium-Term roadmap item that fits cleanly
into a SemVer-minor cut. The two epics are tracked for the v1.1.0 stable
release.

This alpha is published with `npm publish --tag alpha --provenance`; the
`latest` dist-tag continues to point at v1.0.5.

```bash
npm install pdfnative@alpha     # 1.1.0-alpha.1
npm install pdfnative           # 1.0.5 (current stable)
```

## What's new

### 1. Watermark auto-fit (`WatermarkText.autoFit`)

Aggressive presets like `fontSize: 120, angle: -30°` on A4 used to
overflow the page on one or both axes. The watermark renderer now
computes the rotated bounding box

$$\bigl(\,\mathrm{textW}\cdot|\cos\theta|+\mathrm{textH}\cdot|\sin\theta|,\;\mathrm{textW}\cdot|\sin\theta|+\mathrm{textH}\cdot|\cos\theta|\,\bigr)$$

and scales `fontSize` down by

$$\min\!\left(\frac{\mathrm{safeW}}{\mathrm{rotW}},\;\frac{\mathrm{safeH}}{\mathrm{rotH}}\right)$$

when the rotated bbox exceeds the page minus a 24 pt safety margin. The
new field is `autoFit?: boolean`; default is `true` since v1.1.0.
Watermarks that already fit the page are short-circuited and produce
byte-identical output to v1.0.5.

```ts
// New default — auto-fit kicks in when needed
buildPDFBytes(params, {
    watermark: {
        text: { text: 'CONFIDENTIAL DOCUMENT', fontSize: 120, angle: -30 },
    },
});

// Opt-out — preserves exact v1.0.x output, may overflow the page
buildPDFBytes(params, {
    watermark: {
        text: { text: 'CONFIDENTIAL', fontSize: 88, angle: -30, autoFit: false },
    },
});
```

### 2. Unicode ellipsis (`…`, U+2026) in `truncate()`

`truncate()` and the TOC truncation path now use the Unicode horizontal
ellipsis (U+2026) instead of two ASCII dots (`..`) or three (`...`).
The character is mapped to WinAnsi byte `0x85` and renders correctly in
both Latin/Helvetica and CIDFont/Identity-H modes. It is a single
grapheme cluster, ~50% narrower than three dots in Latin mode, and the
typographically correct choice.

### 3. Pixel-based truncation (`truncateToWidth`)

A new exported helper:

```ts
import { truncateToWidth } from 'pdfnative';
truncateToWidth('A long heading that does not fit', 120, 12, enc);
// → 'A long heading…'   (measured in PDF points using `enc`'s metrics)
```

Uses binary search over the encoding context's width function — works
for both Latin (Helvetica metrics) and Unicode (CIDFont width tables).

### 4. Column min/max constraints (`ColumnDef.minWidth` / `maxWidth`)

Two additive optional fields on `ColumnDef`:

```ts
const columns: ColumnDef[] = [
    { f: 0.2, a: 'l', mx: 16, mxH: 16, minWidth: 120 }, // never narrower than 120 pt
    { f: 0.6, a: 'l', mx: 50, mxH: 50, maxWidth: 320 }, // never wider than 320 pt
    { f: 0.2, a: 'r', mx: 14, mxH: 14 },
];
```

Constrained columns are clamped first, then the surplus or deficit is
redistributed across unconstrained columns proportional to their `f`
weight. **When neither field is set on any column, output is
byte-identical to v1.0.5.**

### 5. Four new PDF decode filters in the parser

The parser now handles every standard non-Flate stream filter from
ISO 32000-1 §7.4:

| Filter | Section | Aliases | Notes |
|---|---|---|---|
| ASCIIHexDecode | §7.4.2 | `AHx` | Whitespace ignored, `>` terminates, odd trailing nibble padded with 0 |
| ASCII85Decode | §7.4.3 | `A85` | Adobe variant: `z` shorthand, `~>` end-marker |
| LZWDecode | §7.4.4 | `LZW` | 9–12 bit variable-width codes, CLEAR (256) + EOD (257) |
| RunLengthDecode | §7.4.5 | `RL` | 128 = EOD; literal/repeat encoding per spec |

All four are exported from the package root:

```ts
import {
    decodeASCIIHex, decodeASCII85, decodeLZW, decodeRunLength,
    applyDecodeFilter, KNOWN_DECODE_FILTERS,
} from 'pdfnative';
```

A 256 MiB output cap is enforced on every decoder to defend against
zip-bomb-style adversarial streams. Single-filter and multi-filter-chain
dispatch is wired through `PdfReader.decodeStream()`.

### 6. Live version widget for the docs site

A zero-build, zero-dependency widget that fetches the latest published
versions of `pdfnative`, `pdfnative-cli`, and `pdfnative-mcp` from
`registry.npmjs.org` on page load — and surfaces the **transitive
`pdfnative` pin declared by each downstream package** in plain sight, so
visitors can verify the wiring without leaving the page. Mounted on the
homepage, the [CLI playground](https://pdfnative.dev/playgrounds/cli.html),
and the [MCP playground](https://pdfnative.dev/playgrounds/mcp.html). Falls
back to static defaults when the registry is unreachable.

## Breaking changes

This release contains **two** intentional, opt-out-able changes. Neither
affects the public API surface; both are byte-level differences in
generated output that may break exact-byte snapshot tests.

### B1. `truncate()` ellipsis: `..` → `…`

| Input | v1.0.5 | v1.1.0 |
|---|---|---|
| `truncate('Hello World', 7)` | `'Hello..'` | `'Hello …'` |
| `truncate('Hello', 3)` | `'H..'` | `'He…'` |

**Migration:** update assertions to expect `\u2026`. The character is
fully encoded in both Latin (WinAnsi `0x85`) and CIDFont modes. There
is no opt-out for this change because the previous output was
typographically incorrect.

### B2. `WatermarkText.autoFit` default

`autoFit` defaults to `true` since v1.1.0. PDFs whose watermark
parameters caused the rotated bbox to overflow the page will now emit a
smaller `Tf` size in the watermark content stream.

**Migration:** set `autoFit: false` on the `WatermarkText` object to
restore exact v1.0.x bytes. Watermarks that already fit the page produce
byte-identical output to v1.0.5.

```ts
// Restore byte-stable v1.0.x watermark output
{ watermark: { text: { text: 'DRAFT', autoFit: false } } }
```

## Tracked for v1.1.0 stable

These items belong to the v1.1.0 milestone and are deliberately
deferred from this alpha. They are tracked in
[ROADMAP.md](../ROADMAP.md#in-progress).

- **Issue [#28](https://github.com/Nizoka/pdfnative/issues/28)** —
  PDF/A Latin font embedding via Noto Sans Regular + Bold (SIL OFL-1.1).
  Requires atomic object-graph renumbering across `pdf-builder.ts`,
  `pdf-document.ts`, and `pdf-assembler.ts`, plus `helveticaWidth()`
  → embedded-font width replacement under PDF/A modes.
- **Issue [#25](https://github.com/Nizoka/pdfnative/issues/25)** —
  full UAX #9 W1–W7 + N1/N2 + isolates, multi-pass GSUB driver for
  nested LookupType 4 ligatures, USE-lite cluster classification for
  Indic scripts, GPOS MarkBasePos for isolated Arabic harakat.
- **Auto-fit column widths** — content-aware `mx` computation.
- **Cell clipping paths** — `q re W n … Q` per cell.
- **Constant-memory streaming** — page-by-page assembly without
  buffering the full PDF.
